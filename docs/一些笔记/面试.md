# 面试为什么要问那么多平时工作用不到的东西

```
我以前也是这样想的，那些家伙为啥老是面我们实际工作中用不到的东西——直到我自己作为面试官去面试别人。
首先要澄清一个似是而非的问题：什么叫“平时工作用不到的东西“？
我们潜意识里的回答其实是这样的：那些我自己不知道的东西就是平时用不到的——因为我们在实际工作中不可能使用我们自己不知道的技术来解决问题。
因而，”我们不知道“是因，”实际用不到“是果——而实际中我们常常很自负地颠倒因果。
我也是做 PHP 开发的，在实际招聘中我经常问下面这些问题：

PHP7 相对 PHP5 做了哪些改进？
如何求两个数组的交并差集？
你都了解哪些 PSR 规范？
一次 Web 请求的执行步骤？
什么是聚簇索引？什么是索引覆盖？
什么是回表？
你是如何理解面向对象和面向过程的区别的？
面向对象编程都有哪些特性，以及 OOP 的都有哪些设计原则？
你都了解哪些设计模式？
谈谈你熟悉的几个设计模式。
PHP 数组底层的实现机制是怎样的？
什么是乐观锁、悲观锁？

怎么样，是不是感觉都是“平时工作用不到的”？一些求职者答不上来后，的确也反问过我：“难道你们在实际工作中用到这些吗？”那我们就来逐一分析下，这些是不是真的用不到？

第一条：我只是个代码搬运工，只需要实现公司派发的业务，从不考虑性能、不接触服务器维护、不关心 PHP 语言的发展，知道这些有何用？（考查对 PHP 语言的关注程度，我们对自己整天使用的语言的发展一点都不关注是说不过去的）

第二条：万能的 n 层循环搞定一切——啥？PHP 自身已经提供了现成函数？（面试官有时候并不关心一个问题的确切答案，而是关心你的回答本身，比如这题，我并不关心对方能否准确说出函数名字，我要的答案其实是类似“PHP 里面有现成的函数可以实现“——能用 PHP  自带函数解决的，就不要自己写）

第三条：手里一把梭，干就完事，管他什么规范不规范，什么团队协作、接口标准，干我何事？我只管实现业务，其他一切都是浮云。（考察对规范的关注。很多人遇到这题就本能地大谈特谈编码规范，什么驼峰命名、空格、换行，如果这题满分是 10 分，这种回答只能得 1 分，因为这至少说明他根本没进过 PSR 官网，对 PSR 存在认知上的误区——PSR 远不止编码规范。同样，这题面试官根本不关心你对细节的记忆程度）

第四条：我是做后端的，管 Web 请求干毛用——你说什么？分析 Web 请求性能？加 CDN？nginx 调优？加浏览器缓存？干我何事！（我们声称自己是做 Web 开发的，对一次 Web 请求都不能做一个整体的陈述，有点说不过去。这题是相当开放的，不同水平的人有不同的答案，比如本地域名解析缓存、客户端缓存、DNS 解析、CDN 缓存、TCP 乃至网络协议分层结构、负载均衡、fastCGI 协议、fpm 工作模式等，你能想到的都能说）

第五条：我用到的数据库就是增删改查，顺便给某个字段加个索引——啥？回表？索引覆盖优化？没听说过，从来不用！（本题考查对 MySQL 的掌握程度，而且只是稍作深入，实际做索引优化时可以用到的）

第六条：我用的就是面向对象框架啊，你问我面向对象——额，貌似就是创建一个个类而已，开发中经常用啊，new 就完事，管他什么原则不原则。单一职责？开闭？太虚了。（本题考查对 OO 的理解。大部分人并没有理解 OOP 到底是什么东西，只是觉得自己是在用 OO 框架，便认为自己懂得 OO 了，实际是在用 OO 框架做面向过程的开发，业务稍复杂，代码便很难维护。能很好理解 OO 设计原则的，我会额外加分，因为我觉得对于业务程序员来说， OO 思想比数据结构和算法更实用）

第七条：设计模式？貌似只有框架开发才能用到——你问我代码中为啥这么多 if else，我问谁去？业务就是这么复杂嘛！（同上。设计模式和 OOP 一般放一起，考查求职者的系统设计和重构能力。因为对 OO 和设计模式的理解依赖于实践，因而如果面试对象是初级工程师，面试官一般是把这种题目作为纯加分项来问的——答不上来不减分，答得上来会加分；但对于中高级工程师，则是必备技能。）

第八条：谁没事去研究底层，再重申一遍：我是业务程序员！（该题一般是作为加分项来考查的，答不上来没关系，答得上来会加分。另外，面试官问这种问题还有其他意图：想看看求职者在答不出问题时的表现，以及对后面面试过程的影响——一句话就是看看求职者的心态。公司一般不会仅仅根据求职者回答问题的正确率来决定是否录用，还要考虑其沟通、意愿、心态等因素）

第九条：没用过——你问我为啥库存会扣成负数？这个嘛，概率很低，我们并发不高，可以接受。（对于中高级工程师来说，该题是常识题。对于程序员来说，锁是个必须掌握的知识点，对可靠性稍微有点要求的都离不开锁，如下单、支付、积分操作。确实有很多人从没考虑过这些问题，我就见过好多代码在一个 MySQL 大事务里面做各种远程调用，细究下来其实就是一个乐观锁就能搞定的问题）会写几个代码都能叫程序员，但程序员也分三六九等，有些人做出的项目只能叫”能用“、”实现了业务功能“，而对非功能性需求考虑极少（并发、容错、分布式事务），就是因为知识的广度和深度不够。我们当然可以跟面试官说”我一个人做了某个系统“——但说实话，面试官对这个不感冒，因为每个人都可以一个人做一个系统，但每个人做出来的系统的质量却有可能天壤之别。

面试官问你 HTTP 状态码，无法精确回答出每个状态码含义很正常，但如果你能说出 HTTP 状态码的规律性含义（20X、30X、40X 分别代表什么含义），面试官定会对你刮目相看，因为你是站在更高的层次回答他的问题。
面试官问你多继承，是在考察你对面向对象设计的理解。如果你回答不支持，好的面试官会继续追问为什么？绝大多数 OO 语言都不支持多继承，因为多继承会带来很大的复杂性（如果会 Python 的话，可以去了解下）——那么，现实生活中很多貌似需要多继承来解决的问题如何解决呢？答案是用组合代替继承，有个原则叫“组合优于继承”。
如果你面试时能谈这些，说明你的系统设计能力至少不差——否则面试官可能都会质疑你有无系统设计的能力。
另外，我们得琢磨面试官问问题的意图，很多问题是开放式的（虽然看起来是封闭的），不要认为自己无法精准的给出某些细节就一定给不出漂亮的答案——很多题目，更多考查的是求职者的理解和思维逻辑，而不是记忆力。所以，作为程序员的我们，态度要谦逊点，不要为自己的无知找借口，诚然，不同的面试官水平和风格都不一样，但更多地要从我们自己身上找原因。
```
